# Decisions

## 架构决策

### 1. 前后端分离架构
- **决策**：采用 React (前端) + FastAPI (后端) 分离架构
- **原因**：番茄钟需要精确计时，前端计时容易受页面休眠影响，后端计时更可靠
- **日期**：2026-03-29

### 2. 数据持久化方案
- **决策**：JSON 文件存储，不使用数据库
- **原因**：数据量小（任务、甘特图配置），JSON 足够，且便于调试和备份
- **日期**：2026-03-29

### 3. 多用户认证方案
- **决策**：用户名即可登录，无需密码
- **原因**：私人使用，去密码化提升体验
- **日期**：2026-03-29

### 4. 番茄钟状态机
- **决策**：计时逻辑放在后端，前端负责展示和控制
- **原因**：避免浏览器休眠导致计时不准
- **日期**：2026-03-29

### 5. "Last Thing to Do" 二象限机制
- **决策**：只提供可视化参考，不自动选择任务
- **原因**：用户心理习惯"看到选项再决定"，而不是"被告知该做什么"
- **日期**：2026-03-29

### 6. 甘特图层级
- **决策**：支持无限嵌套子任务
- **原因**：任务分解常需要多层细化
- **日期**：2026-03-29

## 功能决策

### 7. 通知机制
- **决策**：浏览器 Notification + 音频双通道
- **日期**：2026-03-29

### 8. 运动计时特殊逻辑
- **决策**：可提前结束，可超时（归零后正计时）
- **日期**：2026-03-29

### 9. Deadline 优先级提升
- **决策**：48小时内 deadline 自动提升到最高重要程度
- **日期**：2026-03-29

### 16. 健康管理机制
- **决策**：
  - 每2个工作番茄后 → 15分钟休闲时间（看视频等）
  - 每4个工作番茄后 → 运动提醒（拳击30分钟/跑步60分钟）
  - 运动完成后 → 30分钟后洗澡提醒
- **日期**：2026-03-29

### 17. 休息类型区分
- **短休息**：10分钟（普通番茄间隔）
- **休闲时间**：15分钟（每2个工作番茄后）
- **长休息**：15分钟（可选）
- **日期**：2026-03-29

## 部署决策

### 10. Tailscale Funnel 部署
- **决策**：使用 Tailscale Funnel 暴露服务
- **原因**：无需配置反向代理，Funnel 自动处理 HTTPS
- **日期**：2026-03-29

### 11. URL 路径
- **决策**：所有页面路径统一在 `/clock` 下，如 `/clock/login`, `/clock/timer`
- **原因**：不占用根路径，多网站共享 443 端口
- **日期**：2026-03-29

### 12. Last Thing to Do 跳转机制
- **决策**：支持从二象限直接跳转到具体任务页面
- **原因**：用户心理习惯"看到选项再决定"，而不是"被告知该做什么"
- **跳转层级**：二象限 → 任务详情 → 子任务甘特图 → 开始番茄钟
- **日期**：2026-03-29

### 13. 多层级甘特图联动
- **决策**：总体甘特图 ↔ 项目甘特图 ↔ 子任务甘特图，层级间自动同步
- **原因**：避免用户手动同步，同一任务的进度在所有视图保持一致
- **同步机制**：番茄钟完成 → 更新任务进度 → 所有甘特图自动反映
- **日期**：2026-03-29

### 14. 黑客松独立甘特图 + 归档
- **决策**：黑客松独占甘特图，完成后归档到历史黑客松
- **原因**：黑客松是独立事件，不应混入日常任务流
- **归档同步**：黑客松完成 → 贡献任务同步到相关项目甘特图 → 总体甘特图
- **日期**：2026-03-29

### 15. 甘特图任务关联
- **决策**：新建任务只有任务管理一个入口，黑客松/日程只选择/调用任务
- **原因**：避免同一任务多处创建导致数据不一致
- **日期**：2026-03-29

### 19. PolarClaw 联动同步方案
- **决策**：通过 `/api/sync/*` REST + SSE 端点暴露用户状态，使用服务级 API Key（X-Sync-Key）认证，PolarClaw 侧存储 Clock 用户名映射
- **原因**：SSE 单向推送比 WebSocket 更轻量、符合「Clock → PolarClaw」的数据流向；服务级 Key 与用户 session 解耦，适合机器间通信；用户名映射放在 PolarClaw memory 中而非 Clock 数据库，因为映射关系由消费方管理更自然
- **日期**：2026-04-14

### 18. PWA 更新与缓存策略
- **决策**：前端采用 Workbox 的 precache + runtime caching 组合，并在主入口手动注册 Service Worker；新版本通过前端提示用户刷新，而不是静默自动切换。
- **原因**：静态资源和字体需要稳定离线命中，`/api` 请求需要保持 NetworkFirst 以优先拿到最新数据，显式刷新提示可以避免计时中的页面被缓存更新突然打断。
- **日期**：2026-04-10
