## 概述

本知识页阐述 Clone 项目中 **reference/** 目录的设计意图与管理规范。该目录专门用于集中存放 Clone 项目的参考项目、业界最佳实践（SOTA）对标材料以及外部技术文档，是项目知识资产的重要组成部分。

通过建立独立的参考资料管理机制，Clone 项目实现了源码与外部引用的有效分离。这种设计不仅保持了项目仓库的清晰结构，还便于开发者快速定位和查阅相关背景资料，同时也为项目评审、知识传承和技术交流提供了便利。

该规范的核心原则是**只读管理**：目录内仅存放参考资料，不包含项目自身的源代码。这种划分确保了参考资料与开发产物的边界清晰，避免了源码与外部材料的混淆。

## 核心要点

### 目录定位

- **目录名称**：`reference/`
- **存放内容**：Clone 的参考项目、SOTA 对标材料和外部技术文档
- **管理模式：** 只读参考资料

### 参考资料类型

根据规范，[[entity-reference-dir|reference/ 目录]]可包含以下类型的参考资料：

1. **开源项目**：Clone 项目参考或对标的开源实现项目
2. **论文**：相关的学术论文、技术白皮书
3. **技术博客摘录**：有价值的技术博客文章摘录或全文

### 大文件处理策略

对于体积较大的参考资料，规范建议采用以下两种处理方式：

| 处理方式 | 适用场景 | 说明 |
|---------|---------|------|
| **[[entity-git-lfs|Git LFS]]** | 需要将文件纳入版本控制的大文件 | 使用 Git Large File Storage 扩展管理大体积文件 |
| **仅保留链接** | 超大文件或二进制资源 | 在目录中仅保存文件链接，不直接存储文件实体 |

### 管理原则

- **只读性**：明确强调仅存放只读参考资料
- **源码分离**：参考资料与项目源代码严格分离，不混置于同一目录
- **可追溯性**：便于追踪每项功能的参考来源和技术渊源

## 知识结构

### 目的

建立规范的参考资料管理机制，实现项目知识的集中存储与清晰组织，避免源码与外部引用材料混淆。

### 方法

- **目录隔离**：设立独立的 `reference/` 目录专门管理外部资料
- **类型限定**：明确列出可存放的资料类型（开源项目、论文、博客）
- **大文件策略**：对大体量文件采用 [[entity-git-lfs|Git LFS]] 或链接引用方式处理

### 原理

通过目录结构的物理隔离实现内容的逻辑分离，只读属性的强化确保参考资料不被意外修改。这种设计遵循了软件工程中的**关注点分离（Separation of Concerns）**原则，使项目的各个组成部分各司其职，降低耦合度，提升可维护性。
